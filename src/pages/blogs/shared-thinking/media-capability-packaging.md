# 把生图和 TTS 包装成能力：两个微应用的一条配置边界

生图和 TTS 原本已经各自有一套工作台。生图页面负责 Prompt、Workflow 和 ComfyUI 产出，TTS 页面负责 Piper、GPT-SoVITS 和 API 服务商。现在要做的事情很小：让这两个微应用可以被前端和外部接入层识别为两个稳定能力。

这件事看起来像是在卡片上加一个齿轮，实际要先把边界说清楚。能力封装不应该重新发明一套服务商配置，也不应该把工作台里的每个参数全部暴露出去。它只需要回答：这个微应用能处理什么请求，以及当前由哪个已有服务商连接承担。

## 能力不是服务商配置

平台原来已经有完整的服务商连接体系：服务商模板、连接实例、Base URL、API Key、模型同步和角色绑定都在那里。微应用只需要引用连接，不应该再次保存一份凭据。

因此两条能力关系被压缩成了最小结构：

```text
Image Generation Studio
  -> imageGeneration
    -> providerConnectionId

TTS Studio
  -> tts
    -> providerConnectionId
```

`imageGeneration` 和 `tts` 表示能力意图，不是具体的 API 协议，也不是一个可以直接执行的工具。真正的服务商连接仍然由模型设置和服务商配置模块负责。

能力绑定弹框因此只展示可用服务商和当前绑定状态。用户点击确定后，系统只保存 `providerConnectionId`。模型继续使用已有的 `imageGeneration` 或 `voice` 角色配置，API Key、Base URL 和模型选择不会在微应用里重复出现。

这条边界很重要。如果微应用自己再保存一份 API Key，服务商页面和微应用页面迟早会出现两套互相矛盾的事实；如果能力绑定把具体模型也复制一份，角色模型切换又不会及时反映到微应用。能力层只做引用，配置层才拥有配置事实。

## 入口卡片只承担发现和绑定

入口卡片现在有两个动作：进入工作区，或者点击齿轮配置能力。卡片不负责承载完整的调试表单，也不负责解释服务商协议。

```text
Image Generation Studio       [进入工作区] [齿轮]
TTS Studio                    [进入工作区] [齿轮]
```

如果没有能力绑定，卡片只显示一个小徽章：`未配置`。完成绑定后不再增加一段服务商说明，用户需要具体配置时进入齿轮弹框，需要调试参数时进入工作区。这样入口页承担的是状态发现，而不是把三个页面的内容堆到一起。

## 生图继续复用现有模型角色

生图能力的外部字段只有 `imageGeneration`。它不包含 Prompt、Negative Prompt、尺寸、Seed 或 Workflow JSON。那些字段属于生图工作台的请求和调试边界。

服务商选择也有一个前置条件：只有声明支持 `imageGeneration` 角色的连接，才会出现在绑定列表中。绑定成功后，实际使用的模型仍由现有的生图角色配置决定。

这样，外部接入层可以先根据能力名找到生图入口，再由运行时读取当前服务商和角色模型。能力名保持稳定，服务商更换和模型升级不会要求外部调用方同步改协议。

## TTS 不是只有一种运行时

TTS 的难点在于它同时包含本地运行时和远程服务商，但对外仍然需要一个统一能力名：`tts`。

当前内部运行时可以分成三类：

```text
tts
  ├─ Windows 内置语音
  ├─ Piper 本地运行时
  ├─ GPT-SoVITS 本机 Gradio 服务
  └─ API 服务商 voice 模型
```

Windows 内置语音和 Piper 是本地执行路径。Piper 只内置运行时，不打包语音模型；用户自己选择系统语音，或者提供 `.onnx` 与对应的 `.onnx.json`。GPT-SoVITS 也不由应用托管服务进程，页面只调用用户已经启动的本机服务。

API 服务商则复用已有 `voice` 角色模型。它可能走 OpenAI-compatible speech 协议，也可能走特定厂商的语音接口，但这些差异留在 TTS 运行时内部，不向能力调用方扩散。

这就是统一能力的价值：调用方只表达“把这段文本变成音频”，而不是先判断应该调用 Piper、Gradio 还是某家远程语音协议。

## 参考音频是用户素材，不是临时文件

GPT-SoVITS 的参考 WAV 有一个很实际的历史问题。早期流程把文件留在前端 IndexedDB，每次合成都重新上传，后端再生成一个新的参考音频文件。相同的 WAV 被反复使用时，服务器会得到多份重复文件；一键清理也容易让前端缓存和后端文件状态分离。

现在参考音频作为用户素材落入 SQLite BLOB：

```text
上传 WAV
  -> SHA-256 去重
  -> 写入 tts_ref_audios.audio_blob
  -> 返回 refAudioId

再次合成
  -> 只提交 refAudioId
  -> backend 从 SQLite 读取 WAV
  -> 提供给 GPT-SoVITS
```

这不是把音频常驻内存，也不是把它长期展开到媒体目录。SQLite 保存的是可复用的用户素材，生成的音频则仍然落在后端 artifact 目录，属于可以清理的生成产物。

因此一键清理的边界也必须分开：清理生成产物不会删除 GPT-SoVITS 参考素材，参考音频记录可以继续被后续合成使用。

## 历史记录必须尊重真实文件状态

音频任务表里的 `succeeded` 只说明任务曾经成功完成过，不代表当前音频文件仍然存在。生成产物可能已经被清理，或者工作目录发生过迁移。

如果页面切换到 GPT-SoVITS 时，直接拿最近一条 `succeeded` 任务加载预览，就会把一个历史失效任务误当成当前可播放结果。文件不存在时，预览接口应该返回“没有可用历史任务”的事实，而不是抛出 500。

当前处理方式是：成功任务只有在输出文件仍然存在时才进入可用历史列表；预览读取发现文件缺失时返回 404，前端清空预览并保持安静。排队中、运行中和失败任务仍然保留，因为它们承担的是状态展示，而不是可播放历史。

这条规则看似只是一个错误处理细节，实际上是在维护状态语义：数据库记录、文件系统和页面展示必须对“可播放”使用同一个判断。

## 最小能力带来的长期收益

这次封装没有重做两个工作台，也没有新增一套 MCP Tool。它只把已经存在的运行时能力用稳定名称接出来：

```text
能力意图
  -> 已有服务商连接
    -> 已有角色模型
      -> 现有微应用运行时
        -> 本地或远程产物
```

前端可以发现 `imageGeneration` 和 `tts`，服务商配置仍由原来的配置系统维护，微应用页面继续保留各自需要的调试参数，素材和生成产物也拥有不同的生命周期。

真正完成的不是“多了两个按钮”，而是把两个已有工作台放进了同一套能力边界里。以后无论是外部入口、Agent 还是其他微应用需要调用它们，都可以从能力名开始，而不必重新理解每个服务商的配置细节。
